# Infinity AI Turn Controller Sidecar

Status: Sidecar architecture note. Do not implement until explicitly approved.

## Why This Exists

The current Infinity AI graph is improving, but one structural issue remains: simple conversational turns can still require separate LLM calls for classification and response composition.

That is not the final product direction.

The wrong fix would be to add keyword or regex shortcuts for examples like `hi`, `thanks`, or `tell me a joke`. Those examples are symptoms, not requirements. Hardcoding them would recreate the brittle chatbot behavior we are trying to remove.

## Correct Direction

For non-tool conversational turns, replace the separate supervisor/composer path with a single LLM-backed Turn Controller.

The Turn Controller is the conversational brain for one turn. Deterministic code remains the guardrail, validator, executor, and persistence layer.

## Turn Controller Contract

The Turn Controller should return one strict schema containing:

- `conversation_act`
- `active_flow`
- `turn_policy`
- `needs_signal_extraction`
- `needs_tools`
- `needs_recommendations`
- `needs_memory_update`
- `response_draft` or validated response block draft fields
- `should_continue_graph`
- `rationale`
- trace metadata

## Rules

- No hardcoded user-visible assistant prose.
- No keyword/regex/example-based routing for conversational turns.
- No deterministic simulation of examples like greetings, jokes, thanks, cancel, or repeat.
- The LLM decides the conversational act and wording.
- Deterministic code validates schemas, enforces policy, blocks unauthorized tools/memory/recommendations, and persists traces.
- If `needs_signal_extraction=false` and `needs_tools=false`, the graph should stop after the Turn Controller and persist.
- If goal/resource/expert work is needed, the graph continues into the existing deeper flow.
- Platform policy, subscription, booking, payment, usage metering, and actual tool execution remain platform-owned.

## Target Behavior

For ordinary non-tool conversational turns:

- one LLM call maximum
- no fake goal signals
- no expert/resource retrieval
- no memory update unless explicitly allowed and meaningful
- no repair call unless deterministic validation fails after block cleanup
- no hardcoded assistant response

For goal, resource, or expert turns:

- the existing goal-companion flow remains available
- deeper graph nodes run only when intentionally routed
- deterministic scoring and platform policy remain intact

## Acceptance Criteria

- A simple greeting uses one LLM call, not supervisor plus composer.
- Harmless chitchat uses one LLM call, not hardcoded copy.
- Meta/help/repeat/cancel style turns do not create fake signals.
- Goal/resource/expert requests still enter the deeper graph when needed.
- No user-visible prose is generated by deterministic templates.
- Node traces show why the graph stopped early or continued.
- Response blocks remain structured and schema validated.

## Non-Goals

- This is not a multi-agent rewrite.
- This is not a keyword fast path.
- This is not a return to generic chatbot behavior.
- This does not move platform policy or booking decisions into the LLM.
- This should not be implemented until the current provider/router and graph behavior are stable.
